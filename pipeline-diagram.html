Path 1 — Legacy Generate

POST /api/generate/:id
Step 1

User Selects Raga

Frontend calls GET /api/ragas/:id
Loads full raga object from ragas.js (all 14 fields).
↓
Step 2

Build Prompt Payload

buildPayload(raga) composes three parts:

buildStyle() — mood, time atmosphere, tempo, thaat, mode, generic instrument list. Max 1000 chars.

buildPromptText() — Converts swaras → western notes via swaraConverter. Includes scale, aaroha/avaroha as western note names, vadi/samvadi emphasis, alap→gat structure description.

buildNegativeTags() — Excludes vocals, pop/rock/electronic, notes NOT in the raga scale.
Pakad — never included
Aaroha/Avaroha — text mention only
Ornaments — style mention only
Prahar — style descriptors only
Vadi/Samvadi — text emphasis
Scale — fully converted
Difficulty — ignored
↓
Step 3

Suno API — Text-Only Generation

POST /api/v1/generate
Sends prompt + style + negativeTags as natural language. Model V5.
styleWeight: 0.7 · weirdnessConstraint: 0.3 · instrumental: true

No reference audio. Suno interprets text freely — raga theory depends entirely on Suno's understanding of the prompt.
Suno has no raga knowledge — interprets freely
↓
Step 4

Poll → Download → R2 Upload

Polls GET /api/v1/generate/record-info every 5s, max 30 attempts.
Downloads MP3 tracks on SUCCESS.
Uploads to Cloudflare R2 + updates tracks-metadata.json.

Path 2 — Authentic Generate

POST /api/generate/:id/authentic
Step 1

Load Raga + Parse Request

Loads raga from DB. Reads instruments[], duration, genre, useAIPrompt from request body. Falls back to suggestGenres() if no genre.
↓
Step 2

Melody Generation

Generates a note-by-note melody sequence following raga rules. Two sub-paths:
AI Path — OpenAI
AIMelodyGenerator with gpt-4o-mini

Sends to AI: aaroha, avaroha, vadi, samvadi, pakad, special rules

Structure: Alap (35%) → Jor (35%) → Jhala (30%)

Ornaments:
· Meend → 6-step pitch glide
· Gamak → oscillation pairs
· Kan → grace note (15/85 split)

Validation: Checks aaroha/avaroha compliance — warns only, doesn't reject
Pakad — requested, not validated
Aaroha — soft warn
Ornaments — micro-steps, no pitch bend
Prahar — not used in timing
Vadi — 3x weight + velocity
Algorithmic Fallback
AlapGenerator — 6 fixed phases

lower (30%) → ascending (15%) → middle (25%) → upper-touch (15%) → descending (10%) → return (5%)

Weighted random note selection: vadi 3x, samvadi 2x.
Soft aaroha/avaroha enforcement — falls back to all notes if no valid choice.
Pakad — not generated
Aaroha — soft, falls back
Ornaments — none
Prahar — ignored
Vadi — weighted 3x
↓
Step 3

MIDI Build

buildMidi(events) — Fixed 45 BPM. GM instrument 104 (Sitar).
Converts note events to MIDI ticks. No CC events, no pitch bend (meends become staccato steps). Uploaded to R2.
Ornaments degraded — meend glides become discrete steps
↓
Step 4

WAV Render + Upload

renderToWav(events) — Sine waves only at 44.1kHz 16-bit PCM. Simple attack/release envelope (no ADSR). No harmonics, no timbre. Amplitude from velocity.

This is a pitch-accurate reference, not the final output. Uploaded to Cloudflare R2 as reference for Suno.
Pitch accurate, but timbre is meaningless sine waves
↓
Step 5

Cover Prompt + Suno Upload-Cover API

Builds prompt describing the raga, selected instruments, arrangement instructions, and ornament guidance.

Two prompt sub-paths:
· Template buildCoverPrompt() — detailed but verbose (~1500 chars)
· AI generateSunoPrompt() via OpenAI — creative but may hallucinate raga details, truncated to 500 chars

POST /api/v1/generate/upload-cover
Sends R2 WAV URL + prompt + style + negative tags. audioWeight: 0.6 balances reference audio vs prompt.
Pakad — not in cover prompt
Aaroha/Avaroha — not in cover prompt
Ornaments — described in text, Suno interprets freely
audioWeight 0.6 — Suno may reshape melody
↓
Step 6

Poll → Download → R2 Upload

Same polling mechanism as Legacy path.
Polls every 5s, max 30 attempts (2.5 min timeout).
Downloads final MP3, uploads to R2, updates metadata JSON.
⚠ Same Failure Risk
No retry on network errors. No exponential backoff. No circuit breaker.
A single fetch failed kills the entire pipeline after all previous steps succeeded.
